验证依赖安装：


如果遇到其他依赖安装问题，可以尝试清除缓存后重新安装：

## 配置 Nginx

### 1. 创建 Nginx 配置文件

在本地项目目录中创建 Nginx 配置文件：

```bash
# 在本地项目目录中创建 nginx 配置目录
mkdir -p config/nginx
```

创建 config/nginx/online-doc.conf 文件，内容如下：

```
server {
    listen 80;
    server_name 114.55.138.4;  # 使用服务器IP或域名
    
    # 前端静态文件
    location / {
        root /var/www/online-doc/frontend/dist;
        index index.html;
        try_files $uri $uri/ /index.html;
    }
    
    # 后端API代理
    location /api {
        proxy_pass http://localhost:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
    
    # WebSocket代理
    location /ws {
        proxy_pass http://localhost:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "Upgrade";
        proxy_set_header Host $host;
    }
}
```

### 2. 将Nginx配置文件部署到服务器

在服务器上执行以下命令：

```bash
# 将本地配置文件复制到服务器
scp config/nginx/online-doc.conf root@114.55.138.4:/etc/nginx/sites-available/online-doc

# 在服务器上启用配置
ssh root@114.55.138.4 "ln -s /etc/nginx/sites-available/online-doc /etc/nginx/sites-enabled/ && \
rm /etc/nginx/sites-enabled/default && \
nginx -t && \
systemctl restart nginx"
```

## GitHub CI/CD 配置

为了实现自动化部署，我们可以使用 GitHub Actions 来设置 CI/CD 流程。下面是配置步骤：

### 1. 在服务器上创建部署密钥

```bash
# 在服务器上生成SSH密钥对
ssh-keygen -t ed25519 -C "github-actions-deploy" -f ~/.ssh/github-actions-deploy -N ""

# 将公钥添加到authorized_keys
cat ~/.ssh/github-actions-deploy.pub >> ~/.ssh/authorized_keys

# 查看私钥(需要添加到GitHub Secrets)
cat ~/.ssh/github-actions-deploy
```

### 2. 在 GitHub 仓库中添加 Secrets

在 GitHub 仓库页面，进入 Settings -> Secrets and variables -> Actions，添加以下 Secrets：

> **注意**：Secret 名称只能包含字母数字字符([a-z], [A-Z], [0-9])或下划线(_)，不允许有空格，且必须以字母([a-z], [A-Z])或下划线(_)开头。

添加以下 Secrets：

- `SERVER_HOST`: 114.55.138.4
- `SERVER_USERNAME`: root
- `SSH_PRIVATE_KEY`: 上一步生成的私钥内容
- `SERVER_DEPLOY_PATH`: /var/www/online-doc

添加步骤：

1. 点击"New repository secret"按钮
2. 在"Name"字段中输入 Secret 名称（如`SERVER_HOST`）
3. 在"Value"字段中输入对应的值
4. 点击"Add secret"按钮保存
5. 重复上述步骤添加所有必要的 Secrets

### 3. 创建 GitHub Actions 工作流文件

在本地项目根目录创建 `.github/workflows` 目录和 `deploy.yml` 文件：

```bash
# 在本地项目目录中创建工作流目录
mkdir -p .github/workflows
```

编辑工作流文件 `.github/workflows/deploy.yml`：

```yaml
name: Deploy Online Doc

on:
  push:
    branches: [ main ]
  workflow_dispatch:

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'
          
      - name: Setup PNPM
        uses: pnpm/action-setup@v2
        with:
          version: latest
          run_install: false
          
      - name: Get pnpm store directory
        id: pnpm-cache
        run: |
          echo "::set-output name=pnpm_cache_dir::$(pnpm store path)"
          
      - name: Setup pnpm cache
        uses: actions/cache@v3
        with:
          path: ${{ steps.pnpm-cache.outputs.pnpm_cache_dir }}
          key: ${{ runner.os }}-pnpm-store-${{ hashFiles('**/pnpm-lock.yaml') }}
          restore-keys: |
            ${{ runner.os }}-pnpm-store-
            
      - name: Install dependencies
        run: pnpm install
        
      - name: Build frontend
        run: pnpm build:frontend
        
      - name: Build backend
        run: pnpm build:backend
        
      - name: Deploy to server
        uses: appleboy/scp-action@master
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USERNAME }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          source: "frontend/dist/,backend/dist/,package.json,pnpm-lock.yaml,pnpm-workspace.yaml"
          target: "${{ secrets.SERVER_DEPLOY_PATH }}"
          strip_components: 0
          
      - name: Restart application
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USERNAME }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd ${{ secrets.SERVER_DEPLOY_PATH }}
            pnpm install --prod
            pm2 reload ecosystem.config.js
```

### 4. 提交并推送工作流文件到 GitHub

```bash

git add .github/workflows/deploy.yml
git commit -m "Add GitHub Actions workflow for CI/CD"
git push
```

## 系统维护

### 1. 设置自动备份

创建备份脚本:

```bash
cat > /root/backup.sh << EOL
#!/bin/bash
TIMESTAMP=\$(date +"%Y%m%d%H%M%S")
BACKUP_DIR="/root/backups"

# 创建备份目录
mkdir -p \$BACKUP_DIR

# 备份MongoDB数据
mongodump --out=\$BACKUP_DIR/mongodb_\$TIMESTAMP

# 备份项目代码
tar -zcf \$BACKUP_DIR/online-doc_\$TIMESTAMP.tar.gz /var/www/online-doc

# 保留最近7天的备份
find \$BACKUP_DIR -type d -name "mongodb_*" -mtime +7 -exec rm -rf {} \;
find \$BACKUP_DIR -name "online-doc_*.tar.gz" -mtime +7 -exec rm -f {} \;
EOL

chmod +x /root/backup.sh
```

添加到 crontab 定时任务:

```bash
(crontab -l 2>/dev/null; echo "0 2 * * * /root/backup.sh") | crontab -
```

## 故障排查

### 1. 检查服务状态

```bash
# 检查Nginx状态
systemctl status nginx

# 检查MongoDB状态
systemctl status mongod

# 检查Node.js应用状态
pm2 status

```

### 2. 常见问题解决方案

- **前端无法访问**: 检查Nginx配置和状态
- **API请求失败**: 检查后端服务是否正常运行，检查PM2日志
- **WebSocket连接失败**: 
  - 检查防火墙设置，确保端口开放
  - 检查前端环境变量中的`VITE_WS_URL`是否正确配置为服务器地址
  - 检查Nginx配置是否正确代理WebSocket连接
  - 查看浏览器控制台错误信息，常见错误是WebSocket尝试连接到localhost而不是服务器地址

### 3. 解决WebSocket连接错误

如果在浏览器控制台看到类似以下错误：
```
WebSocket connection to 'ws://114.55.138.4:3001/document-xxx' failed
```

这可能是由以下原因导致：

1. **防火墙阻止WebSocket连接**（最常见原因）：
   
   确保服务器防火墙允许3001端口的TCP连接：
   ```bash
   # 检查当前防火墙规则
   sudo ufw status
   
   # 如果3001端口未开放，添加规则允许该端口
   sudo ufw allow 3001/tcp
   
   # 重新加载防火墙规则
   sudo ufw reload
   ```
   
   如果使用阿里云、腾讯云等云服务器，还需要在云服务控制台的安全组中添加入站规则，允许3001端口的TCP连接。
   
   阿里云安全组配置步骤：
   1. 登录阿里云控制台
   2. 进入ECS实例详情页
   3. 点击"安全组" -> "配置规则"
   4. 添加入方向规则：协议类型TCP，端口范围3001/3001，授权对象0.0.0.0/0

2. **Nginx配置问题**：
   检查Nginx配置文件中的WebSocket代理设置是否正确：
   ```
   # 确保有正确的WebSocket代理配置
   location /ws {
       proxy_pass http://localhost:3001;
       proxy_http_version 1.1;
       proxy_set_header Upgrade $http_upgrade;
       proxy_set_header Connection "Upgrade";
       proxy_set_header Host $host;
       proxy_read_timeout 300s;  # 增加超时时间防止长连接断开
       proxy_send_timeout 300s;
   }
   ```

3. **环境变量配置**：
   确保前端项目的`.env`文件中正确配置了WebSocket URL：
   ```
   # 在本地项目的frontend/.env文件中
   VITE_API_URL=http://114.55.138.4:3001/api
   VITE_WS_URL=ws://114.55.138.4:3001
   ```
   
   修改后需要重新构建前端并部署：
   ```bash
   cd /var/www/online-doc
   pnpm build:frontend
   # 然后通过CI/CD部署或手动部署
   ```

4. **后端WebSocket服务未正常运行**：
   检查后端服务日志：
   ```bash
   pm2 logs online-doc-backend
   ```
   
   确保后端服务正确监听WebSocket连接：
   ```bash
   # 检查是否有进程监听3001端口
   netstat -tulpn | grep 3001
   ```

5. **CORS问题**：
   确保后端的CORS设置允许来自前端域名的WebSocket连接：
   ```javascript
   // 在backend/src/index.ts中
   app.use(cors({
     origin: ['http://114.55.138.4', 'https://your-domain.com'],
     credentials: true
   }));
   ```

6. **连接超时**：
   如果WebSocket连接尝试建立但超时，可能是网络延迟问题。可以尝试增加连接超时时间：
   ```javascript
   // 在前端的CollaborativeEditor.vue中
   provider = new WebsocketProvider(
     import.meta.env.VITE_WS_URL || 'ws://localhost:3001', 
     `document-${props.documentId}`, 
     ydoc,
     { 
       params,
       connect: true,
       maxBackoffTime: 10000 // 增加最大重连等待时间
     }
   )
   ```

7. **直接使用端口访问而不通过Nginx代理**：
   如果上述方法都不起作用，可以尝试直接使用3001端口连接WebSocket，绕过Nginx代理：
   ```
   # 修改前端环境变量
   VITE_WS_URL=ws://114.55.138.4:3001
   ```
   
   并确保服务器防火墙和云服务安全组都允许3001端口的访问。

实施上述解决方案后，重启服务并刷新浏览器页面，WebSocket连接问题应该得到解决。
